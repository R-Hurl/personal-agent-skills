# .NET Code Review Skill

You are performing a thorough code review of uncommitted changes in a .NET 10 / C# project. Your goal is to catch logic errors, anti-patterns, and non-idiomatic code — and report findings in `codereview.md` at the project root.

**Start by reading both reference files** — they contain the full checklists with C# examples and MS Docs references:
- `references/dotnet-best-practices.md` — async/await, DI, null safety, LINQ, thread safety, MVC controllers, minimal APIs
- `references/unit-testing-best-practices.md` — test naming, AAA structure, Moq and NSubstitute patterns, async test correctness

---

## Step 1: Discover uncommitted changes

```bash
git status
git diff HEAD          # all uncommitted changes (staged + unstaged)
git diff --staged      # staged-only (run if user says "staged changes")
```

If there are **no uncommitted changes**, say so clearly and stop. Do not fabricate findings.

For each changed `.cs`, `.csproj`, or `appsettings*.json` file:
- Read the **full file contents** — the diff shows what changed, but the full file shows whether it's correct in context.

---

## Step 2: Look up best practices (when uncertain)

The reference file covers the most common patterns. When you hit an edge case you're uncertain about, use the Microsoft Learn MCP tools:

- `microsoft_docs_search` — quick concept lookup
- `microsoft_code_sample_search` — idiomatic examples
- `microsoft_docs_fetch` — full article for deep guidance

Skip the lookup for well-known issues (async void, `.Result`, service locator) — you already know the answer.

---

## Step 3: Apply the checklist

Review each changed `.cs` file against the categories in `references/dotnet-best-practices.md`. Not every category applies to every file — focus on what's relevant to the change.

Severity levels:
- **CRITICAL** — likely causes bugs, crashes, memory leaks, or deadlocks at runtime
- **WARNING** — incorrect or unsafe pattern that could cause problems under load or edge cases
- **SUGGESTION** — valid but non-idiomatic C#; a cleaner way exists

Only report real issues. Do not pad with trivial style nits or hypothetical problems.

---

## Step 4: Write codereview.md

Write (or overwrite) `codereview.md` in the project root:

```markdown
# .NET Code Review

**Reviewed:** <ISO timestamp>
**Branch:** <current branch>
**Changed files reviewed:** <comma-separated list>

---

## Summary

| Severity   | Count |
|------------|-------|
| CRITICAL   | N     |
| WARNING    | N     |
| SUGGESTION | N     |
| **Total**  | **N** |

---

## Findings

### CRITICAL

#### `FileName.cs` — Short description
**Context:** Brief snippet or line reference locating the problem.
**Issue:** What is wrong and why it matters at runtime.
**Fix:** 1–5 lines of idiomatic corrected code.
**Reference:** <MS Docs URL>

---

### WARNING

#### `FileName.cs` — Short description
...

---

### SUGGESTION

#### `FileName.cs` — Short description
...

---

## Clean files

The following changed files had no issues identified:
- `FileName.cs`

---

*Generated by dotnet-code-review skill*
```

Rules:
- Omit a severity section entirely if there are zero findings at that level.
- If there are **no findings at all**, write "No issues found." in the Findings section.
- The **Fix** should show idiomatic .NET 10 / C# — primary constructors, pattern matching, `await Task.WhenAll`, etc. where appropriate.
- Include a **Reference** URL for CRITICAL and WARNING findings wherever you consulted the MS docs or the reference file cites one.

---

## Step 5: Report to user

After writing `codereview.md`:
- State total issue count and breakdown by severity.
- Give the path to the report.
- If there are CRITICAL issues, name the single most important one to fix first.

Do not dump the full report into the conversation — `codereview.md` is the deliverable.
